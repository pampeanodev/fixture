# Group view: Real / Projected standings toggle

## 1. Problem

The group standings table shows a single table built from the `"hybrid"` score
source (`FixtureContext.tsx`): the real result where one exists, otherwise the
user's prediction. That table is the *projection* — it advances the whole group
forward from predictions, so every team reads `PJ = 3` even when only one or two
matches have actually been played.

There is no way to see the **real** current table — the standings counting only
matches that have actually been played. A user looking at Group H during the
group stage can see "where the group is heading" but not "where it stands right
now".

## 2. Goals

- Let the user see the **real** current standings (only played matches) in the
  Group view, alongside the existing **projected** standings.
- Work identically in both `compact` and `expanded` view modes.
- Reuse the existing standings computation — no new ranking algorithm.

## 3. Non-goals

- Out of scope: changing the scoring math, tie-breakers, or best-thirds.
- Out of scope: the knockout/bracket projection, ranking, or simulator — they
  keep their current score sources. This feature only changes what the Group
  view's standings table displays.
- Out of scope: persisting the toggle choice across reloads (localStorage). The
  choice is session/component state only (YAGNI).
- Out of scope: showing both tables at once (stacked / side-by-side). A toggle
  was chosen deliberately — see §5.

## 4. Design

### 4.1 Real standings data source

`calculateStandings(matches, teamIds, "result")` already exists and is already
used in `FixtureContext` (to confirm clinched seeds). The `"result"` source
counts only matches with a real result and ignores predictions — exactly the
"real current table".

Expose a memoized `realStandingsByGroup: Record<string, StandingRow[]>` from
`FixtureContext`, computed symmetrically with the existing `standingsByGroup`
but with a fixed `"result"` source (not the `simulationActive ? "result" :
"hybrid"` source used by the projected memo). It is added to the context value
and dependency array alongside `standingsByGroup`.

### 4.2 Toggle in GroupView

`GroupView` gains a local `useState<"real" | "projected">` defaulting to
**`"real"`**. A two-button segmented control (`Real` | `Proyectada`) renders
above the standings, in **both** the compact and expanded branches.

The selected value picks which array feeds the existing render path:

| Selection | Standings array | Meaning |
|---|---|---|
| `real` (default) | `realStandingsByGroup[group]` | only played matches |
| `projected` | `standingsByGroup[group]` | real-or-prediction (current behavior) |

Everything downstream is unchanged: the expanded `<table>`, the
`<CompactStandings>` component, the qualify / maybe-qualify row coloring (top 2 /
3rd by position), and the `✓` clinched marker driven by `confirmedTeamIds`
(`groupSeedsConfirmed`) all consume whichever array is selected.

Because `GroupView` stays mounted while the user switches groups (only the
`group` prop changes), the toggle choice persists across group navigation within
a session. A full page reload resets it to `real`.

### 4.3 Simulation

During simulation the projected memo already uses `"result"`, so the two tables
coincide. The toggle stays visible but is effectively a no-op in that mode. This
is acceptable and needs no special-casing.

### 4.4 i18n

Add ES keys for the control, then propagate to EN/PT via the project flow
(`add-locale-key` + `i18n-translator`); the locale-consistency test enforces all
three locales are present. Proposed keys (final names left to implementation):

- `groups.standings.source.real` → "Real"
- `groups.standings.source.projected` → "Proyectada"
- optionally `groups.standings.source.label` for an a11y label on the control.

## 5. Trade-offs

- **Toggle vs. stacked vs. side-by-side.** Chose the toggle. The real and
  projected tables have *different* `PJ` per team, so they don't align row for
  row — stacking or side-by-side invites confusion and doubles vertical/
  horizontal space, and side-by-side collapses to stacked on mobile anyway. The
  toggle has the smallest footprint and behaves identically in compact and
  expanded modes. Cost: only one table is visible at a time.
- **Default Real vs. Projected.** Chose `real`. The request was to surface "the
  real current standings"; defaulting to it puts "what actually happened" first,
  with the projection one tap away. (This changes the default the user sees on
  opening a group; the projection is still fully available.)
- **Context memo vs. compute-in-component.** Chose a context memo
  (`realStandingsByGroup`) for symmetry with `standingsByGroup` and to keep the
  computation testable and memoized once for all groups, matching the existing
  pattern.

## 7. Implementation notes

- **Touch points:** `FixtureContext.tsx` (add `realStandingsByGroup` memo +
  context value/type), `GroupView.tsx` (toggle state + control + array
  selection), i18n bundles (es/en/pt).
- **Component-size limit:** `GroupView` is ~144 lines (under the 200 soft
  limit). If the toggle markup pushes it close, extract a
  `GroupStandingsToggle` sub-component under `src/components/group/` rather than
  growing the file (enforced by `.claude/hooks/check-component-size.sh`).
- **Tests:** the data path is just `calculateStandings(..., "result")`, already
  covered by `standings` tests. Add a focused test only if the context memo
  shape warrants it; otherwise verify via `pnpm run build` + manual smoke per
  project convention.
- **No backend / sync impact:** purely a local display concern.
