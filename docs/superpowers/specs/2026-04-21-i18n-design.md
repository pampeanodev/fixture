# i18n — Diseño

**Fecha**: 2026-04-21
**Alcance**: internacionalización completa de la app (UI + equipos + tour + metadatos) a ES/EN/PT
**Estado**: aprobado, pendiente plan de implementación

## Resumen

La app hoy está 100% en español, con strings hardcodeados en components (`.tsx`), tour (`src/tour/steps.ts`), nombres de equipos (`src/data/teams.ts`), y formato de fecha/hora inconsistente (mezcla de `es-AR`, `en-US`, y maps manuales en `utils/formatDate.ts`).

Este diseño introduce una capa i18n in-house, tipada, sin dependencias nuevas, que soporta español, inglés y portugués. Incluye data model change (quitar `name` del `Team`), unificación de formateo de fechas, persistencia local, auto-detección del browser, y un switcher visible en el SidebarFooter.

No se sincroniza el locale entre integrantes de una sala Nostr: es una preferencia per-device.

## Decisiones tomadas

| # | Pregunta | Decisión |
|---|---|---|
| 1 | Idiomas v1 | ES + EN + PT |
| 2 | Alcance | UI + equipos + tour + metadatos (HTML + PWA) |
| 3 | Selección de idioma | Auto-detect del browser + switcher visible |
| 4 | Persistencia | `localStorage`, clave `fixture.locale` |
| 5 | Sync con Nostr | No — preferencia per-device |
| 6 | Librería | In-house (Context + bundles TS tipados); sin deps nuevas |
| 7 | Pluralización | `xxxOne` / `xxxOther` via `Intl.PluralRules` (ES/EN/PT usan la misma regla) |
| 8 | Rich text (tour) | Component `<Trans>` con placeholders `<1>...</1>` |
| 9 | Formato de fecha | `Intl.DateTimeFormat` con BCP47 por locale; se elimina `utils/formatDate.ts` |
| 10 | BCP47 de `es` | `es-AR` (consistencia con la identidad del proyecto) |
| 11 | Formato 12h/24h | 24h para ES/PT, 12h para EN |
| 12 | Switcher UI | En `SidebarFooter`; muestra código ISO (ES/EN/PT), sin banderas |
| 13 | PWA manifest | Manifest único en inglés neutral (no se traduce dinámicamente) |

## Arquitectura

### Módulos nuevos

```
src/i18n/
├── locales/
│   ├── es.ts          # source of truth (objeto `as const`)
│   ├── en.ts          # `satisfies Messages`
│   └── pt.ts          # `satisfies Messages`
├── keys.ts            # tipo MessageKey derivado de typeof es
├── detect.ts          # navigator.language → "es" | "en" | "pt"
├── format.ts          # formatMatchDate, formatTime, formatNumber
├── LocaleContext.tsx  # Provider + useLocale()
├── Trans.tsx          # rich text component para tour/onboarding
└── index.ts           # barrel
```

### Flujo de datos

1. `main.tsx` monta `<LocaleProvider>` alrededor de `<App />`.
2. `LocaleProvider` al inicializar:
   - Lee `localStorage["fixture.locale"]`; si es inválido o no existe, ejecuta `detectLocale()`.
   - Persiste la decisión.
   - Carga sincrónicamente el bundle activo (los 3 bundles pesan ~15 KB combinados; no justifica lazy loading).
   - Sincroniza `document.documentElement.lang`, `document.title`, y `<meta name="description">`.
3. Expone por Context: `{ locale, setLocale, t, formatDate, formatTime, formatNumber }`.
4. `setLocale(next)`: actualiza state, persiste en localStorage, resincroniza metadatos.

### API del hook

```ts
const { t, locale, setLocale, formatDate } = useLocale();

t("topbar.mode.predictions");                       // simple
t("groups.title", { group: "A" });                  // interpolación
t("ranking.points", { count: 12 });                 // pluralización (one/other)
<Trans id="tour.overview.welcomeBody"
  components={{ 1: <strong />, 2: <em /> }} />      // rich text
```

### Type-safety

- `Messages = typeof esMessages` (bundle ES como fuente de verdad del tipo).
- `en.ts` y `pt.ts` usan `const en = { ... } as const satisfies Messages` — TS rompe el build si falta o sobra una key.
- `MessageKey` es el tipo recursivo `Path<Messages>` que produce todos los paths válidos (`"topbar.mode.predictions"`, etc.).
- Overload de `t()` para el caso dinámico de equipos: `t(\`teams.${TeamId}\`)` queda tipado gracias a que `TeamId` es un union literal derivado de `TEAMS`.

### Pluralización

Convención: una key que pluraliza se declara como `xxxOne` y `xxxOther`. El helper elige en base a `count` usando `Intl.PluralRules`:

```ts
// Dentro de t()
if ("count" in vars) {
  const rule = new Intl.PluralRules(BCP47[locale]).select(vars.count as number);
  const suffixedKey = `${key}${rule === "one" ? "One" : "Other"}`;
  // ...lookup suffixedKey
}
```

Para ES/EN/PT sólo hay `one` y `other`. Si en el futuro se agrega un idioma con más categorías (ar/ru/pl), se extiende el set a `zero/one/two/few/many/other` sin cambiar los call sites.

### Rich text (tour)

Los textos largos del tour tienen markup tipo negritas. Usamos un component `<Trans>` que parsea placeholders numerados `<1>...</1>` y los reemplaza con elementos React definidos en code:

```tsx
// tour key: "tour.overview.welcomeBody"
// "Predecís los 104 partidos. <1>3 pts</1> exacto · <2>1 pt</2> solo ganador."
<Trans id="tour.overview.welcomeBody" components={{ 1: <strong />, 2: <strong /> }} />
```

Impl: ~30 LOC, regex `/<(\d+)>([^<]*)<\/\1>/g`, reemplazo por `React.cloneElement(components[n], {}, content)`. Los wrappers (negrita, énfasis, etc.) los define el caller en código — la traducción no puede inyectar HTML crudo. Inmune a XSS por construcción.

## Equipos: data model change

**`src/data/teams.ts` queda agnóstico de idioma:**

```ts
// antes
{ id: "MEX", name: "México", flag: "🇲🇽", group: "A" }

// después
{ id: "MEX", flag: "🇲🇽", group: "A" }
```

El campo `name` sale del tipo `Team`. Todos los call sites que hoy leen `team.name` migran a `t(\`teams.${team.id}\`)`.

**Alternativa descartada**: `name: { es, en, pt }` como objeto. Rechazado porque acopla el modelo de datos al set de idiomas soportados y rompe el invariante "datos estáticos no saben del idioma".

**Type-safety extra**: `teams.consistency.test.ts` verifica en CI que para cada `TEAMS[i].id` exista una entrada en cada locale — cubre el gap que TS no captura (el tipo no enumera los IDs obligatorios).

## Formato de fecha/hora/números

### Estado actual (problemático)

- `utils/formatDate.ts`: maps manuales `SHORT_WEEKDAYS_ES` y `MONTHS_ES` aplicados a output de `Intl.DateTimeFormat("en-US", ...)`.
- `components/ScheduleView.tsx`: mezcla `toLocaleDateString("es-AR", ...)` y `toLocaleTimeString("en-US", ...)`.
- `components/SimulatorView.tsx`: `toLocaleString()` sin locale explícito (sale del browser).

### Estado objetivo (`src/i18n/format.ts`)

```ts
const BCP47: Record<LocaleCode, string> = {
  es: "es-AR",
  en: "en-US",
  pt: "pt-BR",
};

export function formatMatchDate(utcDate: string, locale: LocaleCode): string {
  const date = new Date(utcDate);
  const datePart = new Intl.DateTimeFormat(BCP47[locale], {
    weekday: "short", day: "numeric", month: "short",
  }).format(date);
  const timePart = new Intl.DateTimeFormat(BCP47[locale], {
    hour: "2-digit", minute: "2-digit", hour12: locale === "en",
  }).format(date);
  return `${datePart} · ${timePart}`;
}

export function formatNumber(n: number, locale: LocaleCode): string {
  return new Intl.NumberFormat(BCP47[locale]).format(n);
}
```

Se exponen por el Context para que los componentes llamen `formatDate(match.dateUtc)` sin tocar `locale` directamente.

**`src/utils/formatDate.ts` se elimina** — los maps manuales ya no son necesarios; `Intl` en navegadores 2026 devuelve abreviaciones regionales correctas.

## Detección, persistencia, switcher

### `detect.ts`

```ts
const SUPPORTED = ["es", "en", "pt"] as const;
const DEFAULT: LocaleCode = "es";

export function detectLocale(): LocaleCode {
  for (const lang of navigator.languages ?? [navigator.language]) {
    const primary = lang.toLowerCase().split("-")[0];
    if ((SUPPORTED as readonly string[]).includes(primary)) {
      return primary as LocaleCode;
    }
  }
  return DEFAULT;
}
```

### Persistencia

- Clave: `"fixture.locale"` en `localStorage`.
- Lectura: `useState` con lazy init en el Provider.
- Escritura: en `setLocale()` antes del `setState`.
- Validación: si el valor storeado no está en `SUPPORTED`, se descarta y se cae a `detectLocale()`.

### Switcher UI (`src/components/LocaleSwitcher.tsx`)

- Ubicación: `SidebarFooter` (al lado de Donate/Account).
- Render: botón que muestra el código activo ("ES"/"EN"/"PT"); al clic abre un popover con las 3 opciones mostrando el nombre nativo ("Español"/"English"/"Português").
- Sin banderas (una bandera no mapea 1:1 a un idioma).
- ~40 LOC incluyendo CSS.

## Metadatos

### `<html lang>`, `<title>`, meta description

`LocaleProvider` sincroniza en un `useEffect` cada vez que el locale cambia:

```ts
useEffect(() => {
  document.documentElement.lang = locale;
  document.title = t("meta.title");
  document.querySelector('meta[name="description"]')?.setAttribute("content", t("meta.description"));
}, [locale, t]);
```

`index.html` mantiene valores en español hardcoded para el first paint (antes de que React monte); `LocaleProvider` los sobrescribe en el siguiente tick. Trade-off aceptado: ~50ms de "flash" si el locale detectado no es ES.

### PWA manifest

Manifest único en inglés neutral (`"World Cup 2026 — Fixture & Prode"`). Razón: el nombre del launcher se cachea al momento de instalar la PWA; traducirlo dinámicamente no cambia el ícono del home screen. Bajo ROI.

## Testing

Cuatro tests en `src/i18n/__tests__/`:

1. **`locales.consistency.test.ts`**: tree-walk recursivo comparando `en` y `pt` contra `es`. Falla con mensaje claro (`Missing key "rooms.create.nameLabel" in en`) si falta/sobra una key.
2. **`teams.consistency.test.ts`**: por cada `TEAMS[i].id`, verifica que exista `teams[id]` en los 3 locales.
3. **`detect.test.ts`**: mockea `navigator.languages` con varios arrays, valida mapping y fallback.
4. **`format.test.ts`**: snapshot por locale de una fecha fija (ej. `2026-06-11T20:00:00Z`) para detectar regresiones en BCP47 o `Intl.DateTimeFormat`.

Además, cada fase del rollout mantiene `pnpm test` + `pnpm lint` + `pnpm run build` pasando.

## Rollout — fases

La migración toca ~25 archivos. Se divide en 5 PRs atómicos:

### Fase 1 — Infraestructura

- Crear `src/i18n/` completo (locales, context, format, detect, Trans, switcher).
- ES tiene todas las keys proyectadas; EN/PT son **copias exactas de ES** (placeholder).
- Wrappear `<App />` en `<LocaleProvider>`.
- Ningún componente consume el hook todavía.
- Tests de consistencia + detect + format pasando.
- **Resultado visible**: ninguno.

### Fase 2 — Migrar componentes y tour

- Reemplazar strings hardcodeados por `t()` en todos los `.tsx` y en `tour/steps.ts`.
- Reemplazar callers de `utils/formatDate.ts` por `formatMatchDate` del context.
- Eliminar `src/utils/formatDate.ts`.
- **Resultado visible**: app sigue en español (EN/PT son copias de ES), pero todos los strings pasan por `t()`.

### Fase 3 — Data model de equipos

- Quitar `name` del tipo `Team` y del array `TEAMS`.
- TypeScript rompe en ~15 call sites; arreglar con `t(\`teams.${team.id}\`)`.
- Agregar `teams.consistency.test.ts`.
- **Resultado visible**: ninguno (los nombres ES siguen en `teams.ARG`, etc. de cada locale).

### Fase 4 — Traducciones EN y PT reales

- Llenar `en.ts` y `pt.ts` con traducciones reales de todas las keys.
- Activar el switcher visible en `SidebarFooter`.
- QA manual de los 3 idiomas en desktop y mobile.
- **Resultado visible**: la app es trilingüe.

### Fase 5 — Metadatos

- Sincronizar `<html lang>`, `document.title`, `<meta name="description">` desde `LocaleProvider`.
- Validar en DevTools que cambien con el switcher.
- **Resultado visible**: el browser reporta el idioma correcto para screen readers y herramientas de traducción.

Cada fase es commit/PR independiente y reversible.

## Out of scope

- Sync del locale por Nostr (decisión: preferencia per-device).
- Traducción dinámica del manifest PWA.
- Traducción del `README.md` y `docs/playing-guide.md` (son docs del repo, no parte del app).
- Librería de extracción de mensajes (ej. `i18next-parser`) — el tree-walk del test de consistencia cumple esa función para este volumen.
- Soporte de RTL (árabe/hebreo) — fuera del alcance de los 3 idiomas v1.

## Riesgos y mitigaciones

| Riesgo | Mitigación |
|---|---|
| Olvidar traducir una key nueva | `locales.consistency.test.ts` falla en CI. |
| Strings hardcoded que se escapan de la migración | Después de Fase 2, `grep` de literales en español sobre `src/components` y `src/tour` — catch manual. |
| Re-renders excesivos al cambiar idioma | El Provider memoiza `t` por `locale`; cambios de locale son eventos raros (1–2 por sesión). |
| First paint en idioma incorrecto | `<meta>` y `<title>` hardcoded en ES en `index.html`; Provider los ajusta en el siguiente tick. 50ms de "flash" aceptado. |
| Traducciones EN/PT de baja calidad | Fase 4 es un PR aparte con QA manual; se pueden iterar después del merge sin bloquear nada. |
