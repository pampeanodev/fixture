# Unified Match View

## 1. Problem

Three UI rough edges surfaced once ESPN auto-sync (f7.2) made real results
arrive on their own:

1. **The points indicator shifts the row.** The `+3/+1/0` indicator renders as
   a trailing flex element that only exists when a match is scored, so its
   appearance pushes the team name and score leftward. Reading a column of
   matches is visually jumpy.
2. **The Resultados mode is now redundant.** Results used to be hand-entered in
   a separate `mode: "results"` toggled from the TopBar. With ESPN as the source
   of truth (f7.2), manual entry is only a rare fallback (API down / breaker
   tripped / grace elapsed), yet the user still has to mentally switch "views"
   to see real results vs. their predictions.
3. **The Ranking view has no context.** `RankingView` shows the *active room's*
   rivals, but it appears both as a standalone sidebar item (with no label
   saying which room) and embedded inside `RoomDetail`. The standalone copy
   reads as if it were global, which it isn't.

## 2. Goals

- A match row whose layout never shifts when a result/indicator appears.
- A single reading surface: predictions and real results visible together, no
  mode toggle. Manual result entry only when the API can't provide one.
- Ranking always shown in the context of the room it belongs to.

## 3. Non-goals

- Out of scope: changing the scoring math, the commit-reveal protocol, or the
  auto-sync conflict model (f7.2 stands).
- Out of scope: a cross-room aggregate ranking. Ranking stays scoped to the
  active room (plus locally-imported JSON rivals when roomless).
- Out of scope: changing who may enter a fallback result or how it propagates
  (existing `isMatchEditable` + admin-push rules are preserved).

## 4. Design

### 4.1 Indicator as a fixed-width column

The match row reserves a constant-width slot at its right edge, always rendered.
When there is no scored result the slot holds a neutral placeholder (`·`); when
a result + prediction exist it holds the colored `indicatorFor(...)` label
(`+4/+3/+2/+1/0`). Because the slot is always present at a fixed width, nothing
in the row reflows. Applies to `CompactMatchRow`, the non-compact
`ScheduleMatchCard`, and the prediction rows in `GroupView` / `BracketView`.

### 4.2 Single unified view (no mode toggle)

Remove the Predicciones/Resultados toggle from `TopBar`. Each match row shows,
simultaneously:

- **Prediction** — editable score inputs until the match locks (`isMatchLocked`).
- **Real result** — a read-only badge, shown once a result exists.
- **Indicator** — the fixed-width column from §4.1.

Row states:

| Match state | Prediction inputs | Result badge | Indicator slot |
|---|---|---|---|
| Pre-lock | editable | hidden | neutral `·` |
| Locked, no result yet | read-only | hidden | neutral `·` |
| Result present | read-only | read-only badge | colored points |
| **Fallback** (breaker tripped or grace elapsed, no API result) | read-only | **editable inline** | neutral `·` |

"Fallback" reuses the existing `isMatchEditable(match, ctx)` predicate
(circuit-breaker tripped, or grace period elapsed with no result). Normal
operation keeps the result badge read-only — ESPN is the source of truth.

`state.mode` is retained **internally only** for the simulator
(`ENTER_SIMULATION` sets `mode: "results"`); the user-facing toggle is gone, and
the main match views render the unified prediction+result row regardless of
`mode` (outside simulation). The `SimulatorView` flow is unchanged.

Reducer impact: `SET_GROUP_SCORE` / `SET_KNOCKOUT_SCORE` currently pick the
target field from `state.mode`. In the unified view the caller knows which field
it is editing, so the score-change handlers become explicit about target
(prediction vs. result) rather than deriving it from `mode`. Exact action shape
is left to the implementation plan; the constraint is that a fallback result
edit writes `result` and a prediction edit writes `prediction`, independent of
any global mode.

### 4.3 Ranking integrated into rooms

- Remove the "Ranking" item from `Sidebar`.
- The ranking lives inside `RoomDetail` (already rendered there) under the room
  name header, giving it unambiguous context.
- Roomless case: the Salas (`RoomList`) screen shows the ranking for the local
  player plus JSON-imported rivals, preserving the pre-Nostr sharing path.
- `ViewTarget` loses the `{ type: "ranking" }` variant; navigation to ranking is
  subsumed by `rooms` / `room`.

## 5. Trade-offs

- **Indicator: fixed column vs. overlay chip vs. row color.** Chose the fixed
  reserved column — keeps the number visible, zero reflow, smallest change.
  The overlay chip risks crowding the score inputs; row-color-only loses the
  point value the user explicitly wanted.
- **Manual entry: inline-in-fallback vs. pure-ESPN vs. hidden-in-Settings.**
  Chose inline-in-fallback — keeps a single surface and a recovery path when
  ESPN is wrong/down, without resurrecting a separate mode. Pure-ESPN removes
  all recourse; Settings-hidden splits the flow.
- **Ranking: integrate vs. label-in-place.** Chose integration — eliminates the
  duplicate, contextless standalone copy rather than papering over it with a
  label.

## 6. Open questions

- [ ] None blocking. Fallback-edit authorship follows current behavior (anyone
      edits locally; only the room creator's edits propagate via Nostr).

## 7. Implementation notes

- **Order:** (1) indicator fixed column — small, isolated, ship first; (2)
  unified row + remove toggle — the bulk; (3) ranking integration — navigation.
- **Component-size limit:** `ScheduleView` and `GroupView` are already over the
  200-line soft limit. Rendering prediction+result together must extract
  sub-components under `src/components/<feature>/` rather than growing those
  files (enforced by `.claude/hooks/check-component-size.sh`).
- **Touch points for the toggle removal:** `TopBar` (toggle UI + `TOGGLE_MODE`
  dispatch), `ScheduleView` / `GroupView` / `BracketView` (`scoreField`
  derivation), `CompactMatchRow` / `ScoreInput` (render both fields). Keep
  `TOGGLE_MODE` / `state.mode` in the reducer for the simulator.
- **i18n:** remove `topbar.mode.*` usage from the toggle; keep any strings the
  simulator still needs. Add strings for the read-only result badge label.
  Update all three locales (es/en/pt) — the `Widen<typeof es>` check enforces it.
- **Tests:** pure logic is thin here (mostly presentational). `indicatorFor`
  already covers the indicator content. Add reducer coverage if the
  score-change actions change shape. Verify the rest via `pnpm run build` +
  manual smoke per project convention.
