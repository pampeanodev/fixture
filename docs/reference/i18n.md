# i18n Reference

Current state of the in-house i18n layer. For the design rationale (why in-house, why these idioms), see [`../superpowers/specs/2026-04-21-i18n-design.md`](../superpowers/specs/2026-04-21-i18n-design.md).

## TL;DR

```tsx
import { useLocale } from "../i18n";

function Foo() {
  const { t, locale, formatDate, formatNumber } = useLocale();
  return <div>{t("common.save")}</div>;
}
```

- 3 locales: `es` (default, es-AR BCP47), `en` (en-US), `pt` (pt-BR).
- `es` is the source of truth; `en` and `pt` must mirror its shape.
- Persistence key: `localStorage["fixture.locale"]`. Auto-detected from `navigator.languages` on first load.
- Switcher lives in [`src/components/LocaleSwitcher.tsx`](../../src/components/LocaleSwitcher.tsx), mounted inside `SidebarFooter`.

## File layout

```
src/i18n/
├── types.ts              # SUPPORTED_LOCALES, LocaleCode, BCP47, NATIVE_NAMES
├── locales/
│   ├── es.ts             # Source of truth. `as const` preserves literal types.
│   ├── en.ts             # `as const satisfies Messages` — shape-checked against es
│   └── pt.ts             # same shape contract
├── keys.ts               # Path<T> recursive type → MessageKey union
├── detect.ts             # navigator.language → LocaleCode
├── format.ts             # Intl-based formatMatchDate / formatTime / formatNumber
├── translate.ts          # createT() — interpolation + Intl.PluralRules
├── LocaleContext.tsx     # LocaleProvider + useLocale hook
├── Trans.tsx             # Rich-text component: <1>...</1> markers → React elements
├── index.ts              # Public barrel
└── __tests__/            # consistency, detect, format, translate, Trans, keys.test-d
```

## Adding or editing a key

1. **Add to `es.ts` first.** It's the source of truth. Nest it under an existing namespace (`common`, `topbar`, `rooms`, `tour`, `teams`, etc.) or create a new namespace if genuinely new.
2. **Add the translated value to `en.ts` and `pt.ts`**, keeping the identical path and placeholder shape. TS will refuse to build if you miss one — the `as const satisfies Messages` check catches missing/extra keys.
3. **Use it**: `t("namespace.keyName")` from a component that has `useLocale()`.

The consistency test ([`src/i18n/__tests__/locales.consistency.test.ts`](../../src/i18n/__tests__/locales.consistency.test.ts)) also runs a runtime tree-walk — if TS somehow lets a mismatch through, `pnpm test` fails.

## Interpolation

Variables use `{name}` placeholders:

```ts
// es.ts
title: "Grupo {group}"

// component
t("groups.title", { group: "A" }); // → "Grupo A"
```

Unknown variables leave the placeholder literal (`{unknown}`) so the bug is visible.

## Pluralization

For a quantity that affects wording, write two keys: `fooOne` and `fooOther`. Call with `{ count }`:

```ts
// es.ts
pointsOne: "{count} punto",
pointsOther: "{count} puntos",

// component
t("ranking.points", { count: 1 }); // → "1 punto"
t("ranking.points", { count: 5 }); // → "5 puntos"
```

Under the hood, `Intl.PluralRules(BCP47[locale]).select(count)` picks between `"one"` and `"other"`. For ES/EN/PT this is sufficient; if you add a language with more plural categories (ar, ru, pl → zero/few/many), extend the switch in [`translate.ts`](../../src/i18n/translate.ts) to cover the extra suffixes — call sites stay unchanged.

The consistency test also verifies every `xxxOne` has a matching `xxxOther`.

## Rich text (`<Trans>`)

When a translated string contains inline elements (e.g., bold spans inside a sentence), use numbered placeholders and render with `<Trans>`:

```ts
// es.ts
welcomeBody: "Predecís los 104 partidos. <1>3 pts</1> exacto · <2>1 pt</2> ganador.",
```

```tsx
import { Trans } from "../i18n";

<Trans template={t("tour.overview.welcomeBody")}
       components={{ 1: <strong />, 2: <strong /> }} />
```

The wrapper element is defined in code (the translation file has no HTML). Unknown ids render as plain text.

**Driver.js tours are the exception.** `driver.js` wants HTML strings for `description`; `Trans` returns React. So [`src/tour/steps.ts`](../../src/tour/steps.ts) has a tiny helper `renderTourHtml()` that replaces `<N>...</N>` with `<strong>` + escapes the content. Use it only inside `steps.ts`.

## Team names

Team IDs are a literal union `TeamId` (derived from `TEAM_IDS` in [`src/data/teams.ts`](../../src/data/teams.ts)). Names are translations keyed by ID:

```ts
// es.ts
teams: { ARG: "Argentina", BRA: "Brasil", /* ...48 total */ }

// component (Team.id is TeamId, so the template literal type-checks)
t(`teams.${team.id}`); // → "Argentina" | "Brasil" | ...
```

No `as any`, no casts. The consistency test ([`teams.consistency.test.ts`](../../src/i18n/__tests__/teams.consistency.test.ts)) enforces that every `TEAMS[i].id` has a name in every locale, so if you add a 49th team, the test tells you where to add the translations.

## Date/time/number formatting

Always use the hook's helpers, not `toLocaleString` directly:

```tsx
const { formatDate, formatTime, formatNumber } = useLocale();

formatDate(match.dateUtc);     // "lun 11 jun · 20:00" (es-AR)
                               // "Mon, Jun 11 · 08:00 PM" (en-US)
                               // "seg, 11 de jun. · 20:00" (pt-BR)
formatTime(match.dateUtc);     // Time only
formatNumber(12345);           // "12.345" (es/pt) or "12,345" (en)
```

Formatters are memoized per locale at module scope in [`format.ts`](../../src/i18n/format.ts) — constructing `Intl.DateTimeFormat` is expensive, re-calling is cheap.

**12h vs 24h** is set per-locale: `hour12: locale === "en"`. Change in one place if policy changes.

## The `Widen<T>` / `Messages` type

```ts
// src/i18n/locales/es.ts (tail)
type Widen<T> = T extends string
  ? string
  : T extends readonly (infer U)[]
    ? Widen<U>[]
    : { -readonly [K in keyof T]: Widen<T[K]> };

export type Messages = Widen<typeof es>;
```

Why: `typeof es as const` narrows every string to its literal (e.g. `"Guardar"`). `satisfies Messages` against that type would reject `"Save"` for EN. `Widen<T>` recursively replaces every literal with `string`, so what `satisfies` enforces is **shape** (same keys, same nesting), not **identity** (same values). This is the whole reason the translation contract compiles.

If you see a cryptic TS error from `satisfies Messages` (like "Type 'string' is not assignable to type '\"Guardar\"'"), the `Widen<T>` helper is broken or the `Messages` export got replaced.

## Adding a new language

1. In [`src/i18n/types.ts`](../../src/i18n/types.ts) add the code to `SUPPORTED_LOCALES`, the BCP47 mapping to `BCP47`, and the native name to `NATIVE_NAMES`.
2. Create `src/i18n/locales/<code>.ts` with `export const <code> = { ... } as const satisfies Messages;`. Use another locale as scaffold (copy en.ts or es.ts, translate).
3. In [`src/i18n/LocaleContext.tsx`](../../src/i18n/LocaleContext.tsx) add the new bundle to the `BUNDLES` map.
4. In [`src/i18n/detect.ts`](../../src/i18n/detect.ts) the existing logic already matches any ISO primary code listed in `SUPPORTED_LOCALES` — no changes needed.
5. Consistency tests will tell you if you miss a key; TS will tell you if the shape is wrong.
6. If the new language has plural categories beyond `one`/`other` (ar, ru, pl), extend [`translate.ts`](../../src/i18n/translate.ts) to handle `zero`/`few`/`many` suffixes.

## Gotchas

- **`react-refresh/only-export-components` lint errors in `LocaleContext.tsx` and `Trans.tsx`** are accepted — same pattern as `FixtureContext.tsx` / `NostrContext.tsx`. Don't split the provider/hook into separate files just to silence the rule.
- **`t()` fallback**: if a path doesn't exist, `t` returns the key itself (e.g. `"foo.bar"`). This is intentional so dev-time bugs are loud but not crashy. Don't add a silent fallback to an empty string.
- **`count` must be a number**. `t("foo.points", { count: "5" })` with a string is a no-op on pluralization (the plural branch has `typeof vars.count === "number"`). It will silently render the base key's lookup, which doesn't exist for plural bases — so you get the key echoed back. If you see `"ranking.points"` rendered in the UI, this is why.
- **Tour keys live under `tour.*`** with one sub-namespace per tour (`tour.overview`, `tour.groups`, etc.). Each step uses `Title`/`Body` pairs. Keep this shape when adding new steps.
